# 京華城案言詞辯論 15 場逐字稿＋分析網站｜通盤架構規劃

## 0. 文件定位

本文件是「京華城案言詞辯論 15 場逐字稿＋分析」網站的通盤架構規劃。

預設部署環境：
- GitHub 公開 repository
- GitHub Pages 靜態網站部署
- 無後端伺服器
- 以手機、桌機雙端可讀性為首要目標
- 後續預留 r3f（react-three-fiber）攻防動態圖表入口

本文件處理的是：
- 資訊架構
- 頁面分層
- 導航邏輯
- 內容模型
- 靜態站部署考量
- 後續擴充路徑

本文件不處理：
- 視覺稿定稿
- 前端元件逐一實作
- r3f 圖表細部互動規格
- 法律實體內容判斷

---

## 1. 專案目標

### 1.1 核心目標

建立一個可公開閱讀、可順讀、可跳讀、可圖讀的靜態網站，承載以下三類內容：

1. **15 場逐字稿**
2. **每場分析內容**
3. **通盤分析與攻防視覺化入口**

### 1.2 使用者要能完成的事

使用者進站後，至少要能完成以下三種閱讀任務：

- **順讀**：從第 01 場一路看到第 15 場
- **跳讀**：直接跳到特定場次、特定爭點、特定分析段落
- **圖讀**：先看整體攻防結構，再回到單場文本

### 1.3 成功標準

此網站若要算成功，至少要達成：

- 手機上讀長篇逐字稿不迷路
- 桌機上可同時看導航、正文、段落定位
- 每場分析與逐字稿能共存，而非互相擠壓
- 通盤分析與單場分析有明確接口
- 日後加入 r3f 圖表時，不需重做整站路由與資料模型

## 2. 網站本質判定

這個網站**不是單純文章網站**，也**不是純資料庫介面**。

它的正確定位是：

> **以 15 場法庭攻防為主軸的可導航閱讀系統**

因此架構優先序如下：

1. 可定位
2. 可跳轉
3. 可對照
4. 可順讀
5. 視覺化增益

若順序反過來，容易出現兩種錯誤：
- 只做成 15 篇長文章，導致無法對照與總覽
- 先做很重的互動圖，卻犧牲文字主體的可讀性

## 3. 部署前提與技術約束

GitHub Pages 是靜態網站託管服務，可直接發布 HTML / CSS / JavaScript 靜態檔，也可用靜態站生成流程或 GitHub Actions 建置發布。官方文件也指出，若需要 Jekyll 以外的自訂建置流程，建議使用 GitHub Actions；GitHub Pages 網站屬公開可存取頁面，因此不應放入敏感資料。citeturn675189search1turn675189search2turn675189search11

基於這個前提，本站應採以下原則：

- **前端靜態輸出**：不依賴後端 API 即可完整閱讀
- **內容預編譯**：逐字稿、分析、結構資料在 build 時就進站
- **客端互動為輔**：搜尋、標籤篩選、圖表高亮等可在前端完成
- **路由需考慮 GitHub Pages 子路徑**：若使用 project site，需預留 base path 設定
- **公開 repo 視角寫法**：站內需清楚標註逐字稿性質，避免被誤認為官方逐字稿

### 3.1 建議技術路線

首選：
- **React + Vite + TypeScript**
- Markdown / MDX 或結構化 JSON 作為內容來源
- GitHub Actions 自動部署至 GitHub Pages

理由：
- 可平衡靜態輸出與互動性
- 日後要接 r3f 最順
- 容易做手機 / 桌機不同布局
- 內容資料與展示層可分離

備選：
- Astro：若更偏內容網站可考慮，但後續 r3f 與較重互動的整合複雜度會稍高

## 4. 資訊架構（IA）

### 4.1 第一層路由

建議網站主路由如下：

- `/`：首頁
- `/overview`：通盤分析
- `/battle-map`：整體攻防視覺化入口
- `/sessions`：15 場總列表
- `/sessions/01` ~ `/sessions/15`：單場頁
- `/about`：網站說明／資料來源說明／逐字稿技術說明

### 4.2 第二層概念分區

整站可分成四個功能區：

#### A. 總入口區
處理首次進站者：
- 這個網站是什麼
- 怎麼讀
- 有哪些入口

#### B. 單場閱讀區
處理主體文本：
- 場次摘要
- 分析模組
- 逐字稿
- 本場攻防骨架

#### C. 總體分析區
處理跨場次理解：
- 通盤分析
- 關鍵爭點演變
- 檢辯雙方主軸

#### D. 視覺化區
處理結構導航：
- 全案總圖
- 單場局部圖
- 場次與爭點節點跳轉

## 5. 首頁架構

首頁不要做成單純封面，也不要直接丟長文。

### 5.1 首屏應有三個主入口

1. **通盤分析**
2. **15 場逐字稿與分析**
3. **攻防動態圖表**

### 5.2 中段：15 場進度帶

每一場顯示：
- 場次編號
- 日期 / 上午下午
- 本場主題
- 分析是否完成
- 圖表是否完成

手機版可橫滑，桌機版可整排或分組顯示。

### 5.3 下段：閱讀方式說明

以三種閱讀模式寫清楚：
- 依序讀
- 跳著讀
- 先看結構圖再回文本

### 5.4 全站警語（首頁版）

首頁必須固定出現逐字稿性質說明：

> 本站逐字稿由法庭公開播送內容進行語音轉文字後再整理修訂，非官方人工逐句校正版本，僅供分析與研究參考。正式引用、轉載或確認原意時，請以法院正式資料為準。

這不是裝飾，是整站閱讀前提。

---

## 6. 單場頁架構

單場頁是全站核心。

### 6.1 桌機版布局

建議三欄：

- **左欄**：15 場列表＋全站導航
- **中欄**：主閱讀區
- **右欄**：本場目錄／關鍵點／段落跳轉

### 6.2 手機版布局

建議單欄，但加三個固定導航元件：

- 頂部 sticky 場次條
- 底部浮動「目錄」按鈕
- 分析 / 逐字稿 / 圖表 tab 切換

### 6.3 單場頁固定模組

每一場至少固定有以下區塊：

1. **頁首資訊**
   - 第幾場
   - 日期與時段
   - 主題
   - 上一場 / 下一場

2. **本場一句話摘要**
   - 這場在做什麼
   - 這場在 15 場中的位置

3. **單場警語（頁首版）**
   - 強化提醒逐字稿性質

4. **分析模組**
   - 這場在做什麼
   - 戰略意圖
   - 論證結構
   - 亮點
   - 脆弱點或未竟點
   - 關鍵概念
   - 與前後場連結

5. **本場攻防骨架**
   - 檢方主張
   - 辯方主張
   - 主要交火點
   - 延續至下場的爭點

6. **逐字稿區**
   - 時間戳
   - 講者
   - 逐段文字
   - 可展開 / 收合
   - 可被錨點定位

7. **頁尾跳轉**
   - 上一場 / 下一場
   - 回通盤分析
   - 回總圖

### 6.4 單場警語文案建議

頁首版：

> 本場逐字稿係依公開播送內容進行語音轉文字後，再就人名、法條、機關名稱與法庭用語等進行修訂整理。此版本非官方人工逐句校正稿，僅供分析參考；若需正式引用或確認原意，請以法院正式資料為準。

逐字稿區短版：

> 本逐字稿為語音轉文字後修訂稿，非官方人工校正版本，僅供分析參考；正式引用請以法院資料為準。

## 7. 三種閱讀模式

### 7.1 順讀模式

適用對象：第一次完整閱讀者。

必要元素：
- 上一場 / 下一場
- 場次連續感
- 每場在全局中的位置提示
- 已讀進度可視化（可選）

### 7.2 跳讀模式

適用對象：已知想看哪一場或哪一爭點者。

必要元素：
- 側欄場次清單
- 關鍵詞 / 爭點篩選（第二階段）
- 本場內部目錄
- 可直接跳到分析 / 逐字稿 / 亮點

### 7.3 圖讀模式

適用對象：先看結構再回到文本者。

必要元素：
- 從總圖點入單場
- 單場反向跳回總圖
- 圖表高亮本場在全案中的位置

## 8. 通盤分析頁架構

即使目前「還在製作中」，也不該只放一句施工中。

應先放出**穩定骨架**，讓此頁成為全站的總論接口。

建議章節：

1. 本案 15 場辯論總覽
2. 檢方主軸
3. 辯方主軸
4. 關鍵爭點地圖
5. 爭點如何跨場次演變
6. 場次索引（每節對應可點回單場）

若內容未完成，先顯示：
- 已完成章節
- 預計補寫章節
- 各章節佔位標題

這樣讀者看到的是「可持續擴充的總論頁」，不是空白頁。

---

## 9. 視覺化頁（battle-map）架構

r3f 圖表不應是首頁裝飾，而應是第二套導航系統。

### 9.1 第一層：全案總圖

應至少顯示：
- 15 場節點
- 各場主軸
- 檢 / 辯攻防主線
- 關鍵轉折點
- 延續爭點連線

### 9.2 第二層：單場局部圖

從單場頁可看到該場局部圖：
- 本場主張
- 反駁
- 尚未解決節點
- 與前後場連結

### 9.3 圖表角色定位

圖表的角色是：
- 結構顯影
- 導航
- 對照輔助

不是取代文字主體。

## 10. 內容資料模型

為了靜態站可持續擴充，應先把每場資料結構固定。

### 10.1 Session 基本資料

```ts
interface SessionMeta {
  id: string;              // "01"
  order: number;
  date: string;
  timeOfDay: string;       // 上午 / 下午
  title: string;
  theme: string;
  sideFocus: string;       // 檢方 / 辯方 / 程序
  status: {
    analysis: boolean;
    transcript: boolean;
    graph: boolean;
  };
}
```

### 10.2 單場內容資料

```ts
interface SessionContent {
  summary: string;
  strategicPurpose: string;
  argumentStructure: string[];
  highlights: string[];
  weakPoints?: string[];
  keyConcepts: string[];
  relatedSessions: string[];
  transcriptNotice: string;
  transcript: TranscriptBlock[];
  localGraph?: GraphData;
}
```

### 10.3 逐字稿段落資料

```ts
interface TranscriptBlock {
  id: string;
  timestamp: string;
  speaker: string;
  text: string;
  tags?: string[];
}
```

### 10.4 好處

固定資料模型後，才能穩定做：
- 單場頁
- 場次列表
- 本場目錄
- 關鍵詞篩選
- 爭點圖表
- 交叉引用

若前期直接把每場都寫成自由格式長文，後期會很難加導航與對照功能。

## 12. 可讀性規格

### 12.1 長文閱讀原則

逐字稿是高密度長文，應遵守：
- 段落間距明確
- 時間戳與講者視覺上可快速掃到
- 正文行寬不可過寬
- 手機版字級不可太小
- 分析與逐字稿視覺層級必須明確分離

### 12.2 警語的視覺地位

警語不可只放頁尾。

至少三層都要有：
- 全站層
- 單場頁首層
- 逐字稿區層

### 12.3 逐字稿區互動最低需求

MVP 版至少做到：
- 段落錨點
- 講者標示
- 時間戳
- 區塊折疊

第二版再考慮：
- 引文複製連結
- 關鍵字高亮
- 分析段與逐字稿互跳

## 13. SEO 與公開站資訊倫理

由於此站是公開 GitHub Pages 網站，且逐字稿非官方人工校對版，因此：

### 13.1 每頁 metadata 應明示

- 本站為整理與分析用途
- 逐字稿為語音轉文字後修訂版本
- 非官方人工逐句校正稿

### 13.2 避免誤導性頁標題

不建議寫成：
- 「京華城案官方逐字稿」
- 「法院逐字稿全文」

應寫成：
- 「京華城案言詞辯論第 01 場｜分析與逐字稿整理」
- 「京華城案 15 場逐字稿與分析（非官方人工校正）」

### 13.3 about 頁應固定說明

about 頁建議包含：
- 本站定位
- 逐字稿來源與處理方式
- 文字誤差風險
- 正式引用原則
- 聯絡方式（若需要）

## 15. 先鎖死的關鍵決策

在正式開工前，最值得先鎖死的不是視覺風格，而是以下四件事：

### 15.1 每場分析模組是否固定

建議固定。

否則 15 場之間會很難比較，也很難做視覺化與索引。

### 15.2 逐字稿與分析是否分頁

建議**同頁分區，不同模式切換**。

若完全拆頁，使用者會一直在分析頁與逐字稿頁之間迷路。

### 15.3 內容原始格式

建議：
- 分析用 Markdown / MDX
- 逐字稿用結構化 JSON

### 15.4 圖表資料是否與文字資料共用 session id

建議從一開始就共用。

否則後續文字頁與圖表頁的互跳會變麻煩。

## 16. 本案網站的最終結構判斷

此網站最合理的骨架應是：

> **首頁做入口**
> **單場頁做閱讀主體**
> **通盤分析頁做總論接口**
> **視覺化頁做結構導航**

而不是：
- 首頁塞滿全文
- 15 篇文章各自孤立
- 圖表做成獨立炫技頁

真正的主體仍然是文字，但文字不該以「純長文」方式堆放，而應被整理成可導航的靜態閱讀系統。

## 17. 建議下一步

建議接下來依序做三件事：

1. 先定 **內容 schema**
2. 再畫 **單場頁 wireframe**
3. 最後才做 **實際前端骨架**

若順序反過來，容易先把版型做出來，後面才發現資料結構無法支撐 15 場與全局圖的對照需求。
