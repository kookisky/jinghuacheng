## 2. 人名地名校正流程（A1 + A2）

### 2.0 校正來源優先級

人名校正有三個來源，優先級由高到低：

1. **通盤分析**（`京華城案_通盤分析.md`）：所有出現在通盤分析中的人名都經過人工確認，精準度最高。這是人名的黃金標準。
2. **校正總表**（`京華城案_人名地名校正總表_v3_6.md`）：覆蓋面最廣（200+ 人名），但部分條目標「待確認」。
3. **第14場校正筆記**（`京華城案_第14場_人名校正筆記.md`）：補充校正。

**執行方式：**

步驟一：先從通盤分析中提取所有人名，建立「黃金名單」。這些人名包括但不限於：柯文哲、沈慶京、應曉薇、彭振聲、黃景茂、邵琇珮、吳順民、李文宗、李文娟、張志澄、端木正、廖彥鈞、林俊言、鄭深元、蘇正文、廖威志、唐于智、李傳侯、陸正義、蕭奕弘、謝律師、許英傑、林欽榮、林洲民、朱亞虎、陳佳敏、林青、苗博雅、李文宗、王尊侃、周榆修、邱復生、王令麟、蔡壁如、黃珊珊、劉秀琳、林芝羽、楊智勝、白仁德、范有偉、許芷瑜、蔣萬安等。

步驟二：校正總表中若有同名條目，以通盤分析的寫法為準。若校正總表某條目標「待確認」但通盤分析中明確使用了該名字，則視為已確認，納入自動替換。

步驟三：校正總表中有但通盤分析沒提到的人名（邊緣角色），維持校正總表的原有規則（跳過待確認、跳過大致正確等）。

### 2.1 校正總表結構

校正總表（`京華城案_人名地名校正總表_v3_6.md`）由多個 markdown 表格組成，分布在約 20 個章節中。每張表格的核心欄位是：

- `正確名稱`：替換目標
- `STT常見錯誤`：替換來源（逗號分隔的多個變體）

部分章節還有 `身份`、`備註`、`修正原因` 等欄位，但校正流程只需要「正確名稱」和「STT常見錯誤」。

另有 `京華城案_第14場_人名校正筆記.md` 提供補充校正，格式相同。

### 2.2 提取規則

從校正總表提取替換對照時，遵守以下規則：

**必須跳過的行：**
- `STT常見錯誤` 欄位為 `—`、`（大致正確）`、`（待確認STT錯誤）` 的行
- `正確名稱` 欄位包含 `（待確認）` 的行（整行跳過，不做自動替換）

**提取格式：**

```json
{
  "corrections": [
    {
      "correct": "沈慶京",
      "wrong_variants": ["審清金", "沈清今", "審信金", "省信心", "沈進今", "沈金晶", "陳菁菁", "沈沁英"],
      "context_dependent": false
    },
    {
      "correct": "本院|鈞院",
      "wrong_variants": ["軍院"],
      "context_dependent": true,
      "context_rule": "speaker_role=judge→本院; speaker_role=prosecutor|defense→鈞院"
    }
  ]
}
```

### 2.3 上下文依賴替換

以下替換需要根據講者角色判斷：

| STT 錯誤 | 講者為法官時替換為 | 講者為檢察官/律師時替換為 |
|----------|------------------|----------------------|
| 軍院 | 本院 | 鈞院 |

做法：先跑 parser 拆出逐字稿的 TranscriptBlock（見第 3 節），取得每段的 speaker，再對每段內容跑上下文依賴替換。非上下文依賴的替換可以在 parse 之前先全文跑。

### 2.4 替換執行順序

1. **長詞優先**：先替換較長的錯誤變體，再替換較短的。避免「陳俊源」的變體「陳俊仁」先被「陳俊」的某個規則部分匹配。
2. **全詞匹配**：替換時不要做子字串匹配。「林青」不應該把「林青榮」裡的「林青」換掉。實作建議：對中文人名，檢查替換目標前後是否為標點、空格、或字串邊界。若替換目標是 2 字人名而目標前後緊接的字會構成其他已知人名，跳過此次替換。
3. **冪等性**：替換後的結果再跑一次應該不會再改變任何東西。

### 2.5 替換報告

每場跑完替換後，產出一份報告：

```
第01場：共替換 47 處
  沈慶京：審清金(3)、沈清今(2)
  邵琇珮：邵秀慧(5)、邵秀惠(1)
  ...
未替換（上下文依賴）：軍院(12) → 需 parse 後處理
未替換（待確認）：許淑元(1)
```

### 2.6 替換後不要覆蓋原檔

原始 markdown 保留不動。替換後的版本寫入新目錄（例如 `corrected/`）。

## 3. Markdown → 結構化資料 Parser（A3）

### 3.1 目標

每場原始 markdown 拆成三個檔案：

```
sessions/
  01/
    meta.json        # 場次資訊
    analysis.md      # 分析正文（markdown 原樣保留）
    transcript.json  # 逐字稿段落結構化資料
  02/
  ...
  15/
```

### 3.2 切分邏輯

每場 markdown 的結構是：

```
[標題區]        ← 第 1 行到第一個 ## 之間
[分析區]        ← 從分析正文開始，到「逐字稿技術說明」或「逐字稿全文」標題之前
[技術說明區]    ← 「逐字稿技術說明」或「逐字稿全文」標題到第一個講者行之間
[逐字稿區]      ← 從第一個 **審判長** \[時間戳\] 到檔案結尾
```

**切分步驟：**

1. 找到第一個匹配 `**審判長**` 且同行有 `\[` 的行 → 記為 `transcript_start`
2. 從 `transcript_start` 往上找最近的匹配 `## 逐字稿` 的行（不管是 `## 逐字稿技術說明`、`## 逐字稿全文`、`## **逐字稿全文**`）→ 記為 `notice_start`
3. 從 `notice_start` 到 `transcript_start - 1` 之間的內容 → `transcriptNotice`
4. 從第 1 行到 `notice_start - 1` → 標題區 + 分析區
5. 從 `transcript_start` 到檔案結尾 → 逐字稿區

### 3.3 標題區 → meta.json

從第 1 行提取場次資訊。

**場次編號提取**：用正則 `第(\d{2})場` 從第 1 行抓。

**日期與時段提取**：從前 5 行找匹配 `(\d{2,3})/(\d{1,2})/(\d{1,2})` 或 `12\d{2}` 格式的日期，以及「上午」「下午」。

**注意：各場標題格式不統一**，以下是已知變體：

| 場次 | 第 1 行格式 |
|------|-----------|
| 01 | `# 京華城案法庭直播　第01場分析` |
| 07 | `# **京華城案法庭直播 第07場分析+逐字稿**` |
| 09 | `# 京華城案法庭直播　第09場分析` |
| 12 | `# 京華城案法庭直播　第12場分析` |
| 15 | `# 京華城案 第15場 1224下午 分析與逐字稿` |

不要依賴完整標題格式，只用正則抓場次編號。

**主題提取**：通常在第 3 行（`###` 開頭），包含日期和主題描述。整行保留為 `title` 欄位。

**meta.json schema：**

```typescript
interface SessionMeta {
  id: string;              // "01" ~ "15"
  order: number;           // 1 ~ 15
  date: string;            // "2025-12-15" 格式（民國114年轉西元）
  timeOfDay: "上午" | "下午";
  title: string;           // 完整主題描述
  sideFocus: "檢方" | "辯方" | "混合" | "程序";
  status: {
    analysis: boolean;     // Phase A 全部為 true
    transcript: boolean;   // Phase A 全部為 true
    graph: boolean;        // Phase A 全部為 false
  };
  transcriptNotice: string; // 信心度/技術說明全文
}
```

### 3.4 分析區 → analysis.md

從第 1 行到 `notice_start - 1`，整塊保留為 markdown。

**不要拆分析區的內部結構。**原因：各場分析的章節標題格式不統一（有的用 `##`，有的用 `###`，有的不用標題符號），章節命名也不統一（有的有「基本觀念」，有的沒有）。強拆會增加 edge case 處理量，但收益很低——前端直接 render markdown 就夠了。

唯一需要從分析區提取的結構化資訊是：**章節標題列表**（用於前端右側目錄導航）。提取規則：找所有 `##` 或 `###` 開頭的行，去除 markdown 符號和粗體標記後，作為目錄項目。

### 3.5 逐字稿區 → transcript.json

從 `transcript_start` 到檔案結尾，parse 成 `TranscriptBlock[]`。

**TranscriptBlock schema：**

```typescript
interface TranscriptBlock {
  id: string;              // "01-001", "01-002"... （場次編號-段落流水號）
  timestamp: string;       // "00:13" 或 "1:51:32"（原樣保留）
  timestampSeconds: number; // 換算成秒數，用於排序和錨點定位
  speaker: string;         // 原始講者標記，如 "審判長"、"辯護人（蘇律師）"
  speakerRole: SpeakerRole; // 見下方對照規則
  text: string;            // 段落內容（可包含多個換行和中間時間戳）
  isSectionBreak: boolean; // 此段落前是否有 --- 分隔線
}

type SpeakerRole = "judge" | "prosecutor" | "defense" | "defendant" | "other";
```

**Parse 規則：**

1. **講者行辨識**：匹配 `**XXX** \[時間戳\]` 格式（其中 `\[` 可能是 `\\[` escaped 形式）。這行標誌一個新 block 的開始。
2. **內容收集**：從講者行的下一行開始，到下一個講者行（或 `---` 分隔線、或檔案結尾）之前的所有行，合併為該 block 的 `text`。
3. **中間時間戳**：逐字稿中有大量獨立一行的時間戳（如 `\[05:16\]`），這些不是新的 block，而是嵌在當前 block 內容中的時間標記。兩種處理方式皆可：
   - 方式 A：保留在 text 中，前端 render 時用正則偵測並特殊處理
   - 方式 B：拆成子段落（sub-blocks），每個中間時間戳切一刀
   - **建議用方式 A**（簡單，且前端有更多彈性決定要不要顯示中間時間戳）
4. **分隔線**：`---` 獨立一行表示段落分隔（通常對應休庭或主題切換）。下一個 block 的 `isSectionBreak` 設為 `true`。
5. **時間戳格式**：有兩種——`MM:SS`（如 `00:13`、`51:52`）和 `H:MM:SS`（如 `1:51:32`）。超過一小時的場次會出現後者。`timestampSeconds` 統一換算。

**speakerRole 對照規則：**

```
speaker 包含「審判長」→ judge
speaker 包含「檢察官」→ prosecutor
speaker 包含「辯護人」或「律師」→ defense
speaker 包含「被告」→ defendant
其他 → other
```

注意：部分場次的講者標記格式較複雜，例如：
- `**辯護人（蘇律師）**` → defense
- `**檢察官（廖彥鈞）**` → prosecutor
- `**謝律師（應曉薇辯護人）**` → defense
- `**姚維慈律師（沈慶京辯護人）**` → defense
- `**被告（柯文哲）**` → defendant
- `**被告（沈慶京）**` → defendant

規則是看關鍵詞，不是看格式結構。優先序：
1. 包含「審判長」→ judge
2. 包含「檢察官」→ prosecutor
3. 包含「被告」→ defendant
4. 包含「辯護人」或「律師」→ defense
5. 以上都不匹配 → other

### 3.6 已知格式差異彙整

以下是 5 場樣本（01、07、09、12、15）中觀察到的格式差異，parser 需處理：

| 差異項目 | 變體 | 處理方式 |
|---------|------|---------|
| 第 1 行標題 | 有/無粗體、有/無「法庭直播」、有/無日期編碼 | 只用正則抓場次編號 |
| 分析區章節標題級別 | `##` / `###` / 無標題符號 | 分析區整塊保留 markdown，不拆 |
| 分析/逐字稿分界標誌 | `## 逐字稿技術說明` / `## 逐字稿全文` / `## **逐字稿全文**` | 統一用「找第一個 **審判長**」作為逐字稿起點 |
| 信心度說明 | 獨立章節 / `**信心度說明**：` 一段 | 統一放 `transcriptNotice` |
| 講者名稱複雜度 | 簡單（`**檢察官**`）到複雜（`**姚維慈律師（沈慶京辯護人）**`） | 用關鍵詞匹配 speakerRole |
| 時間戳格式 | `MM:SS` / `H:MM:SS` | 統一換算成秒數 |
| 分析區有/無「基本觀念」章節 | 第 01 場有，其他場不一定有 | 不影響，分析區整塊保留 |

## 4. 前端架構規格（A4 + A5 + A6）

### 4.1 技術選型

- **框架**：React + Vite + TypeScript
- **樣式**：Tailwind CSS
- **路由**：React Router（hash mode，避免 GitHub Pages SPA 路由問題）
- **Markdown render**：react-markdown 或 @mdx-js/react
- **部署**：GitHub Actions → GitHub Pages

### 4.2 路由結構

```
/#/                    → 首頁
/#/overview            → 通盤分析（內容已完稿）
/#/sessions            → 15 場列表頁
/#/sessions/01         → 單場頁
/#/sessions/01~15      → ...
/#/about               → 網站說明
```

### 4.3 內容載入策略

- 15 場的 `meta.json` 打包進主 bundle（體積小，首頁場次列表需要）
- 每場的 `analysis.md` 和 `transcript.json` 用 dynamic import，進入單場頁時才載入
- `transcript.json` 是最大的檔案（每場約 70-150KB），必須 lazy load

### 4.4 單場頁布局

### 4.5 分析/逐字稿 tab 切換實作注意

**不要用 conditional render。**兩個區塊都在 DOM 裡，用 CSS `display: none` / `display: block` 控制顯隱。原因：切換時各自保持滾動位置，不會回到頂部。

## 5. 通盤分析頁規格（A7）

### 5.1 內容來源

通盤分析正文在 `京華城案_通盤分析.md`，約 33KB / 328 行。直接作為 markdown render。

### 5.2 結構

文章分兩大板塊：

**第一部分：總評**（五個結構性論點）

- 二、辯方迴避的七個問題就是案件的七根承重柱
- 三、檢方的排兵布陣

- 五、政治獻金案打穿了京華城案的人設

結論段。

### 5.3 場次超連結

文中所有提到「第01場」「第09場」等場次的地方，前端 render 時應轉為可點擊的超連結，跳轉到對應的 `/#/sessions/01` 等單場頁。

## 7. 開工順序

1. `scripts/parse-corrections.ts`：吃校正總表，產出 `corrections.json`
2. `scripts/apply-corrections.ts`：用 `corrections.json` 對 15 場 markdown 跑批次替換，產出 `corrected/` 目錄
3. `scripts/parse-sessions.ts`：吃 `corrected/` 目錄中的 15 場 markdown，產出 `src/content/sessions/01~15/` 結構化資料
4. 把通盤分析 markdown 放入 `src/content/overview.md`
5. 前端骨架：初始化 Vite + React + TypeScript + Tailwind + React Router
6. 單場頁：用真實資料開發，確認閱讀體驗
7. 通盤分析頁：render markdown + 場次超連結互跳
8. 場次列表頁（含四階段分組）+ 首頁
9. About 頁
10. 部署流程

步驟 1-4 完成後，步驟 5-10 可以用真實資料開發，不需要造假資料。
